Decentralized Blackboard ArchitectureAn Event-Driven, Bossless SLM Swarm with TTL FallbackThis architecture introduces a completely novel paradigm for our final project. Instead of relying on a centralized "Orchestrator" LLM or a synchronous routing pipeline, we treat inference as a decentralized, event-driven microservices environment.Models act as autonomous asynchronous workers that monitor a shared state matrix (the "Blackboard") and bid on tasks based on their compute penalty and confidence.🧠 Why This is Novel for the ThesisStandard architectures (Routing, Speculative Decoding, POA Debate) rely on three rigid assumptions:Synchronous Execution: Models run in a fixed sequence.Centralized Control: A router or arbitrator dictates who answers.Query = Unit of Work: A prompt goes in, a string comes out.We break all three. By adopting an Embedded Systems / RTOS philosophy, tasks are treated as stateful objects (OPEN, IN_PROGRESS, BLOCKED, RESOLVED). The SLMs buzz around handling micro-tasks concurrently, while the monolithic 70B model acts as a "Lazy Heavyweight" that only burns tokens when a task physically stalls out on the board.⚙️ Core Mechanics1. The Blackboard (Shared Memory)Instead of passing strings between functions, queries are posted to a shared dictionary as BlackboardTask objects. Every task has a status and a Time-To-Live (TTL) expiry.2. Autonomous BiddingWhen a task is OPEN, all idle SLMs evaluate it simultaneously.They calculate a quick internal bid using the following equation:$Utility = Predicted\_Confidence - (Cost\_Weight \times Compute\_Penalty)$If the utility exceeds the baseline threshold (e.g., 0.65), the SLM autonomously claims the task, changing its state to IN_PROGRESS.3. Sub-Tasking (The "Active Oracle" pattern)If an SLM gets stuck during generation, it does not fail or return garbage. Instead, it outputs a strict token (SUB_TASK: <query>). The worker loop intercepts this, marks the main task as BLOCKED, and posts the sub-task back to the board. Once the sub-task is resolved, the main task wakes up and resumes.4. The 70B Heavy SweeperThe monolithic model (e.g., Llama 3.3 70B) monitors the board but refuses to bid on fresh tasks due to its massive Compute_Penalty. It operates on a TTL (Time-To-Live) trigger. If a task sits unresolved for $>600ms$, or gets permanently BLOCKED, the 70B model wakes up, solves it brute-force, and goes back to sleep.🛠 Integration & Setup1. File PlacementSave the implementation code as blackboard.py in the architectures/ directory.2. Instantiating in the RunnerIn your ExperimentRunner or main.py, instantiate the architecture by passing your two SLMs (e.g., Qwen 3.5 4B and Llama 3.2 3B) and your heavy LLM (e.g., Llama 3.3 70B).Pythonfrom architectures.blackboard import DecentralizedBlackboardArchitecture
from core.models import OpenAICompatibleModel

# Initialize your vLLM endpoints
slm_primary = OpenAICompatibleModel(model_id="Qwen/Qwen3.5-4B", base_url="http://localhost:8001/v1")
slm_secondary = OpenAICompatibleModel(model_id="meta-llama/Llama-3.2-3B-Instruct", base_url="http://localhost:8003/v1")
llm_sweeper = OpenAICompatibleModel(model_id="meta-llama/Llama-3.3-70B-Instruct", base_url="http://localhost:8000/v1")

# Initialize the architecture
arch = DecentralizedBlackboardArchitecture(
    slm=slm_primary,
    secondary_slm=slm_secondary,
    llm=llm_sweeper,
    cost_weight=0.15,      # Tune this! Higher = heavier penalty for the 70B model
    task_type="mcq"        # or "open"
)

# Run standard interface
response = arch.run(query)
print(f"Final Answer: {response.predicted_answer} | Cost: ${response.cost_usd:.5f}")
🧪 Hyperparameters for BenchmarkingWhen running the empirical analysis for the thesis, tune these specific parameters to shift the efficiency/accuracy frontier:cost_weight (Default: 0.15): Controls the swarm's fiscal conservatism. Increasing this makes the SLMs more aggressive in bidding, keeping the 70B asleep longer.ttl_expiry (Default: 600ms): The grace period before the 70B model intervenes. Lowering this guarantees flat latency curves but burns more tokens. Raising it allows SLMs more time to resolve complex dependency chains (sub-tasks) themselves.Bid Threshold (Default: 0.65): Found in _worker_loop. Adjusting this determines how confident an SLM must be before it dares to claim an OPEN task.📊 Expected Metadata OutputThe Response.metadata object will contain a flat log of the asynchronous inference_steps. Because execution is non-linear, you will see steps interweaved across models, and latency will track the wall-clock time of the async event loop rather than the sum of sequential generation times.
