\chapter{CONCLUSIONS AND FUTURE WORK}

This term report demonstrates that the thesis project has matured into a substantial experimental platform, even though the full validation phase is still in progress. The implemented contribution now includes a shared architecture contract in which the same \texttt{Query} can be processed by different response-generation strategies that all return the same \texttt{Response}, a remote inference topology spanning multiple GPU tiers, MLflow-backed tracking and reporting, and an instrumentation layer that records latency, tokens, cost, energy, and emissions. Together, these components make the project more than a collection of model scripts: they provide a reusable platform for architecture-level comparison under a consistent measurement policy.

The pilot results also support a clear but carefully bounded empirical takeaway: there is no universally superior architecture across all evaluation dimensions. The standalone single-model configuration remains the strongest raw-accuracy operating point in the reported pilots, while routing and other hybrid designs represent viable intermediate operating points by reducing the frequency of large-model invocations and improving parts of the cost-energy profile. At the same time, the pilots show that these savings may come with noticeable latency overhead, which is why EATS should not be read in isolation; it is most informative when interpreted together with accuracy, latency, cost, and energy.

The most important contribution of this term report is therefore not a final leaderboard claim, but the establishment of a repeatable measurement-first infrastructure for comparing orchestration architectures rather than models alone. The current quantitative evidence is promising and already useful for shaping the thesis direction, yet it remains exploratory because the intended coding benchmark, the human preference workflow, and the fully stable end-to-end validation path are not complete enough for final thesis-grade conclusions.

\section{Future Work}

\begin{enumerate}
    \item The planned internal coding benchmark should be finalized as a revision-stage \texttt{custom\_stratified} track with clearly validated easy, medium, and hard tasks, so that the main thesis comparison rests on a stable coding-oriented benchmark rather than on a legacy placeholder.
    \item The experiment runner and benchmark-loader path should be stabilized for repeatable end-to-end runs, including cleaner configuration validation, more explicit registry behavior, and less environment-dependent launch friction during repeated comparisons.
    \item The project-specific \texttt{HumanEval} workflow should be completed as a UI-backed human preference evaluation surface, explicitly preserving that it is not the OpenAI code-generation benchmark, while the blackboard-style and pure-swarm architectures should be validated on proper benchmarks instead of remaining conceptual or pilot-only contributions.
    \item The experimental scope should then be expanded through additional benchmark coverage, routing-threshold calibration, fine-tuned SLM ablations, and stronger cross-architecture comparisons that include standalone, hybrid, ensemble, and swarm operating points under the same reporting policy.
\end{enumerate}
